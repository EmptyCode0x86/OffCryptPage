OffCode Tor Chat: Product overview

Tor Chat (TorChat) lets anyone stand up their own chat on the Tor network: no cloud account, no clearnet bind. Guests open your .onion in Tor Browser; every message is encrypted in the browser with WebCrypto before it reaches the host. ChatServer relays and stores ciphertext only.

Host with Server Manager (section below). Password rooms are server-blind E2EE (the host cannot derive the room key). Open rooms still encrypt in the browser, but the key is derived from the room id: intentional public channels. Hidden rooms stay off the public list and use invite links; add a password for private E2EE. Creators can close a room to new members and recover hidden rooms via a Profile recovery code (My rooms). Optional Full Vanguards hardens long-lived onions against guard-discovery.

How it works

Browser encrypts first: password rooms stay server-blind; open rooms are public by design

Client

Tor Browser (User)

  • Password → PBKDF2 → roomKey + join proof
  • Message → AES-256-GCM → ciphertext only

Plaintext never leaves the browser. Locked-room keys stay client-side; open-room keys are derived from the room id.

Tor Network (.onion)
Host

ChatServer

  • Receives ciphertext only (never plaintext from clients)
  • Stores encrypted payload in SQLCipher DB
  • Broadcasts the same ciphertext to room members
  • Cannot derive locked-room keys; open-room keys = f(room id)

Features

🧅

Tor-only hosting

ChatServer binds 127.0.0.1 only. Tor Expert Bundle publishes a v3 .onion and forwards VirtualPort 80 (and the app port) to localhost; no clearnet chat listener.

  • HiddenServiceVersion 3, ClientOnly 1, SafeLogging 1
  • SOCKS disabled on the HS process (SOCKSPort 0)
  • Open in Tor Browser: http://ADDRESS.onion/
🔐

Browser encryption

WebCrypto PBKDF2 → AES-256-GCM. Separate KDF domains for join proof vs room key on locked rooms so the server cannot derive the message key from join material.

  • AAD binds roomId|senderId|nonce|clientSentAtUnixMs
  • Locked: password stays in the browser: server-blind E2EE (min 8 chars)
  • Open: key from torchat-open-material:{roomId}: public channel by design
  • 1:1 ECDH hybrid + safety number (3×5 decimal) for peer verify / TOFU
  • No Double Ratchet yet; prefer short message retention if passwords may leak
💬

Rooms, invites & TTL

Open, password-protected, and/or hidden rooms with invite links and per-message lifetime.

  • Hidden rooms: not listed; join via ?invite= (+ password if locked)
  • Close room to new members in Settings: blocks all joins (including invites); existing members stay
  • Creator recovery code (Profile) binds your hidden rooms to My rooms: Copy / Restore after Tor New Identity
  • Retention: 30 min … 1 week / Never (default 8 h)
  • Unique senderId per room; server binds send identity to the session
🗄️

Ciphertext storage

SQLCipher database under LocalAppData. DPAPI-wrapped DB key. Parameterized queries. Plaintext is never written to disk.

  • Rooms + ciphertext envelopes only
  • Max 500 messages per room (oldest trimmed)
  • Expired rows: secure_delete + WAL checkpoint / VACUUM (best-effort page wipe)
🛡️

Tor Browser-friendly UI

Copper/Forge web UI designed for onion use: no third-party CDNs, self-hosted fonts, SRI on scripts.

  • Strict CSP (script-src / connect-src 'self')
  • No IndexedDB / WebRTC; manage tokens may use browser storage for creator UI only
  • DOM built with textContent / createElement

🖥️ Server Manager

Windows host console for your Tor chat stack

Server Manager is the host-side control panel: it starts and stops ChatServer plus a Tor v3 Hidden Service, shows the .onion URL to share, optionally enables Full Vanguards, and lets you administer rooms and storage. The host never receives plaintext from clients; password-room keys stay in browsers. Open-room keys are derivable from the room id by design.

  • Start / stop ChatServer and the Tor Hidden Service from one place
  • Copy and share the published .onion URL
  • Optional Full Vanguards checkbox (Python + Stem) for long-lived HS hardening
  • Sets TORCHAT_ALLOWED_ONION for ChatServer CORS when the onion is known
  • Live room list (room id, locked flag, online count only, no message content)
  • Remove a selected room in realtime: connected users are forced back to the lobby; ciphertext for that room is deleted with secure_delete + VACUUM
  • Remove SQL data: securely overwrites (then deletes) the SQLCipher DB, WAL/SHM, and db.key.dpapi; optional checkbox also wipes Tor Hidden Service keys (.onion identity)
  • Generate new address: securely wipes HS keys and publishes a new .onion (chat DB unchanged)
  • Point at your ChatServer project path and load/save host config as needed

🔐 Security Architecture

Defence-in-depth for a Tor Hidden Service chat, every layer is independently hardened

🛡️ Request Protection

# Protection
1 Security headers: CSP (default-src 'self', connect-src 'self'), X-Frame-Options DENY, nosniff, Referrer-Policy no-referrer, Permissions-Policy, COOP, CORP
2 Cache-Control: no-store for HTML and /js/* (sensitive chat shell)
3 Kestrel hardening: AddServerHeader = false, MaxRequestBodySize ≈ 64 KB
4 Rate limiting: Global HTTP ceiling; hub negotiate and /api/* per connection; Create / Join / Send / GetRooms hub method limits (ConnectionId, Tor has no useful client IP)
5 CORS: loopback always; TORCHAT_ALLOWED_ONION whitelist (set by Server Manager). Fallback any *.onion if unset. No AllowCredentials
6 AllowedHosts: 127.0.0.1;localhost;*.onion

🛡 Full Vanguards - Guard-Discovery Protection Optional

A standard Tor Hidden Service chooses its entry (guard) nodes randomly on every circuit build. Over time a network-level adversary can watch which relays the server connects to and gradually narrow down the real server IP. This is called a guard-discovery attack and is especially dangerous for long-lived hidden services.

L2 Vanguards
4 pinned middle-layer nodes, rotated ~1-45 days (addon defaults)
L3 Vanguards
8 pinned nodes, rotated ~1-48 hours (addon defaults)

By pinning layers, only a tiny controlled set of relays ever see the hidden service's circuit-building traffic, making large-scale guard-discovery statistically infeasible.

torrc change Adds ControlPort 127.0.0.1:27551 and CookieAuthentication 1. Without the checkbox the torrc is minimal and no ControlPort is opened.
Python addon Server Manager launches vanguards.py (mikeperry-tor/vanguards) after Tor bootstraps. Requires Python 3.10+ and stem library.
Bandguards Intentionally disabled in shipped vanguards.conf (enable_bandguards = False). Bandwidth-based circuit-killing would disconnect active chat sessions; L2/L3 pinning defence remains fully active.
ControlPort security Loopback-only (127.0.0.1:27551), cookie authentication. Cookie file stays on disk under %AppData%\TorChat_ServerManager\TorData\.
Soft-fail design If Python or stem is missing, the checkbox is silently ignored and the hidden service continues normally. Server Manager log shows the exact reason.

⚠ Vanguards adds latency (~200-500 ms per circuit) due to longer paths. This is a deliberate privacy/performance tradeoff. It does not replace E2EE and does not hide the server from its own guard nodes; it only limits which relays can observe circuit-building over time.

Frequently Asked Questions

What is OffCode Tor Chat?

OffCode Tor Chat is a self-hosted Tor-only realtime chat. Guests open your .onion in Tor Browser; messages are encrypted in the browser with WebCrypto AES-GCM. ChatServer relays and stores ciphertext. Password rooms are server-blind E2EE; open rooms are intentional public channels.

Do I host Tor Chat myself?

Yes. Server Manager on Windows starts ChatServer and a Tor v3 Hidden Service on your machine. Optional Full Vanguards hardens long-lived onions. There is no OffCode-hosted chat cloud; you control the onion URL and the stack.

How does end-to-end encryption work?

Clients encrypt with WebCrypto (PBKDF2 then AES-256-GCM) before sending. Locked rooms: join proof and room key use separate KDF domains (torchat-join-v1 / torchat-e2ee-v1) so the server cannot derive the message key; the password never leaves the browser. Open rooms: still encrypted, but the key is derived from the room id, anyone who knows the id (including the host) can decrypt. 1:1: ECDH hybrid plus a safety number for out-of-band peer verify.

Are open rooms private from the server?

No. Open rooms are public encrypted channels. Use a password (and optionally a hidden room with an invite link) when you need server-blind E2EE.

How does Tor Chat relate to EU Chat Control 1.0?

Chat Control 1.0 lets major messaging providers voluntarily scan private communications. Tor Chat is a different model: self-hosted on Tor, not a centralized big-tech messenger. You avoid that scanned platform path by hosting your own Hidden Service with browser E2EE. This is architectural privacy, not legal advice.

Do clients need Tor Browser?

Yes. Guests open http://ADDRESS.onion/ in Tor Browser. The chat HTTP stack binds loopback only; Tor publishes the Hidden Service. The UI is Tor Browser-friendly: self-hosted fonts, SRI scripts, and no third-party CDNs.

Can the host delete rooms or wipe chat data?

Yes. Server Manager can remove a room in realtime (lobby kick + secure_delete/VACUUM for that room’s ciphertext). Remove SQL data overwrites then deletes the SQLCipher DB and db.key.dpapi (optional: also wipe Tor HS keys / .onion). Generate new address securely rotates the Hidden Service identity. Tor clients cannot call the admin delete API.

What are Close room and the creator recovery code?

Close room to new members (creator Settings) rejects all new joins, including invite links; people already in the room stay. It does not make an open room server-blind; use a password for that. The creator recovery code on Profile binds hidden rooms you create to My rooms; Copy it and Restore after Tor New Identity or clear site data so you can list those rooms again.

Tor Chat not working: what should I check?

If Tor Chat is not working, confirm Server Manager has started ChatServer and Tor, that the published .onion URL is copied correctly, and that guests use Tor Browser (not a normal browser). A Tor error or “Tor not working” on the client usually means Tor Browser cannot reach the network. Restart Tor Browser, wait for a circuit, then reopen http://ADDRESS.onion/.

My .onion address won’t open: how do I fix it?

Ensure the Hidden Service is running in Server Manager, copy the full .onion link again, and open it only in Tor Browser. If the page fails to load, wait for Tor to bootstrap, try a new circuit, and verify the host PC is online with ChatServer still running.
